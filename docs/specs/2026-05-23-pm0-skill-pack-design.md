# PM0 Skill Pack Design

Date: 2026-05-23
Status: Draft for review

## Goal

PM0 is product memory for AI agents. It helps founders and product-minded builders keep durable product context in the repo, then use that context before changing a product surface.

PM0 v1 ships as a public skill/plugin pack for Claude Code, Codex, Cursor, GitHub Copilot, Gemini, OpenCode, and similar agent harnesses.

## Product Shape

PM0 should feel founder-native. Use language like proposal, PRD, product area, build, kill, revise, and ship. Avoid process-heavy terms in v1 unless they are needed for the file contract.

Core objects:

| Object | Meaning |
| --- | --- |
| Project | Whole-product memory and company/product context. |
| Context | Optional supporting memory supplied or requested by the user. |
| Surface | Durable source of truth for one product area or flow. |
| Proposal | Active product thinking artifact for an idea, problem, or change. |
| PRD | Implementation-ready artifact generated during handoff. |
| GitHub CI | Optional PR check against `.pm0` memory. |

Do not make these first-class v1 concepts:

- Decision
- Product Check
- `/pm0 prd`
- `/pm0 review`

## Repository Memory

PM0 writes durable memory into `.pm0/`:

```text
.pm0/
  project.md
  contexts/
  surfaces/
    index.md
    onboarding.md
  proposals/
    2026-05-23-onboarding-empty-state.md
  prds/
    2026-05-23-onboarding-empty-state.md
```

### Project Memory

`.pm0/project.md` stores global product truth:

- product one-liner
- target users
- problem
- business model, if known
- current stage
- important product surfaces
- current goals
- founder/product principles
- assumptions needing confirmation

It should not become a dumping ground for every proposal, transcript, ticket, or PR status.

### Contexts

`.pm0/contexts/` is user-extensible. Users can bring their own context files, and PM0 can create selected context files when useful.

PM0 must not generate every possible context during `/pm0 init`. It should infer or ask what matters for the repo.

Possible context files include:

- `design-system.md`
- `target-users.md`
- `roadmap.md`
- `competitive-landscape.md`
- `product-vision.md`
- `positioning.md`
- `go-to-market.md`
- `mvp-framing.md`
- `success-metrics.md`
- `pricing.md`
- `workflow-analysis.md`
- `research-plan.md`

These are optional. A landing page, dashboard, onboarding flow, developer tool, or pricing project may need different context.

### Surfaces

`.pm0/surfaces/{surface}.md` is the source of truth for a product area.

Surface files should include:

```markdown
# {Surface Name}

## Purpose

## Current Behavior

## Target Users

## Product Principles

## Known Problems

## Metrics

## Active Proposals

## Accepted PRDs

## Rejected Proposals

## Open Questions

## Agent Notes
```

When a proposal is accepted for engineering, rejected, or marked as needing more discussion, PM0 updates the related surface. Future agents should be able to read one surface file and understand the durable product history for that area.

### Proposals

Proposals are date-prefixed:

```text
.pm0/proposals/YYYY-MM-DD-{surface}-{slug}.md
```

A proposal should capture:

- problem or opportunity
- current behavior
- desired outcome
- target users or segment
- evidence and caveats
- proposed scope
- non-goals
- acceptance criteria
- open questions
- build notes
- status: Draft, Investigating, Proposed, Accepted, Rejected, Built, Shipped

Rejected proposals should be summarized under `Rejected Proposals` in the surface file. PM0 should then ask whether to keep or delete the rejected proposal file.

### PRDs

PRDs are generated by `/pm0 handoff`, not by a separate `/pm0 prd` command.

PRDs are date-prefixed:

```text
.pm0/prds/YYYY-MM-DD-{surface}-{slug}.md
```

A PRD should include:

- product intent
- problem
- current behavior
- desired behavior
- users and segments
- evidence
- scope
- non-goals
- acceptance criteria
- metrics and instrumentation
- risks and tradeoffs
- implementation notes
- verification expectations

The related surface should link accepted PRDs under `Accepted PRDs`.

## Skill Packaging

PM0 should follow the Impeccable-style packaging pattern: one skill with command routing and per-command reference files.

```text
skills/
  pm0/
    SKILL.md
    reference/
      init.md
      analyze.md
      discuss.md
      build.md
      handoff.md
      github-ci.md
    scripts/
      scaffold-pm0.mjs
      product-ci.mjs
```

`SKILL.md` should stay lean:

- explain PM0 in one or two paragraphs
- route `/pm0 <command>`
- require reading `.pm0/project.md` and relevant surface memory first
- load only the matching command reference
- explain that repo memory is durable truth and external sources are evidence

Reference files own command-specific behavior.

Implementation should use the local repo examples as packaging references:

- `examples/impeccable-main` for the one-skill command router, per-command reference files, and user-facing command vocabulary.
- `examples/superpowers-main` for plugin manifests, README structure, cross-harness packaging, and test/sync patterns.

Follow their structure and ergonomics. Do not copy PMOS content or templates into PM0.

## Commands

PM0 uses one slash command with subcommands:

```text
/pm0
/pm0 init
/pm0 analyze <surface>
/pm0 discuss <surface-or-proposal>
/pm0 build <proposal>
/pm0 handoff <proposal>
```

Do not use colon syntax such as `/pm0:init`.

### `/pm0`

With no argument, show the command menu and ask what the user wants to do.

### `/pm0 init`

Creates `.pm0/` and makes the repo product-aware.

Reads:

- README and existing docs
- package files
- visible routes/app structure
- existing agent instructions
- product URL if the user provides one
- product context supplied by the user

Writes:

```text
.pm0/project.md
.pm0/contexts/
.pm0/surfaces/index.md
.pm0/proposals/
.pm0/prds/
```

It may create selected context files, but should not generate every context file by default.

It should ask whether to add optional GitHub CI.

### `/pm0 analyze <surface>`

Analyzes a product area as a product surface, not as a code review.

Reads:

- `.pm0/project.md`
- `.pm0/contexts/*` relevant to the surface
- `.pm0/surfaces/{surface}.md`, if present
- relevant routes/components/docs
- accepted PRDs and proposal summaries linked from the surface
- available external evidence only when relevant

Writes or updates:

```text
.pm0/surfaces/{surface}.md
```

The output should explain current behavior, user intent, product constraints, known problems, and open questions.

### `/pm0 discuss <surface-or-proposal>`

Turns a founder idea, complaint, customer quote, or product problem into a proposal. It can also continue an existing proposal after new feedback.

Reads:

- project memory
- relevant contexts
- surface memory
- active proposal, if one is named
- related PRDs and rejected proposal summaries
- current branch/PR state if useful
- host-agent tools and integrations when the question needs external evidence

Writes:

```text
.pm0/proposals/YYYY-MM-DD-{surface}-{slug}.md
```

`discuss` should route evidence intelligently. The skill should ask or inspect what tools and integrations the current host agent has available, then choose only relevant sources.

Examples:

| Product question | Likely sources if available |
| --- | --- |
| Login is failing | Intercom/Zendesk support tickets, Sentry, GitHub issues, recent PRs, auth logs. |
| Users drop during onboarding | PostHog/Amplitude funnels, session replay, support tickets, onboarding surface memory. |
| Pricing confusion | Support tickets, sales notes, checkout analytics, competitor pages, pricing surface memory. |
| Recent product regression | Git history, GitHub PRs/issues, Sentry, release notes. |
| Competitor/market question | Web research, competitor sites/docs, sales notes. |

Rules:

- Do not dump raw tickets, emails, transcripts, or replay data into `.pm0`.
- Store links, short summaries, counts, and caveats.
- Mark evidence strength and contradictions.
- If evidence is weak, say so.
- Do not use every available integration just because it exists.

At the end, PM0 should ask whether the user wants to continue discussing, build the proposal, reject it, or hand it off.

### `/pm0 build <proposal>`

Builds the smallest useful product change that justifies the proposal.

This does not make PM0 a coding agent. The host agent implements the change using PM0 memory as product context.

Reads:

- proposal
- linked surface memory
- project memory
- relevant contexts
- relevant code files
- current branch/PR state

Writes:

- code changes through the host agent
- optional proposal build notes
- optional PR description draft

Rules:

- preserve product constraints from the surface file
- prefer small, reversible changes
- do not silently expand scope
- if a branch/PR should be created, suggest it before large edits

### `/pm0 handoff <proposal>`

Closes the product loop. It generates a PRD and updates durable surface memory.

Possible outcomes:

- accepted for engineering
- rejected
- needs more discussion

Reads:

- proposal
- linked surface
- project memory
- relevant contexts
- branch diff or PR description, if present
- stakeholder or user feedback supplied by the user
- host-agent tools and integrations if needed for final evidence

Writes:

- `.pm0/prds/YYYY-MM-DD-{surface}-{slug}.md` for accepted handoffs
- updated `.pm0/surfaces/{surface}.md`
- optional GitHub PR description update
- optional proposal status update or deletion

If rejected:

- do not generate a full PRD unless the user asks
- summarize the rejection under `Rejected Proposals` in the surface file
- ask whether to keep or delete the proposal file

If accepted for engineering:

- generate the PRD
- link the PRD from `Accepted PRDs` in the surface file
- update current behavior, principles, metrics, or agent notes if the proposal changes durable surface memory
- optionally update the GitHub PR description if a PR exists

If more discussion is needed:

- do not generate a PRD
- keep the proposal active
- update the proposal with the remaining questions or missing evidence

## Optional GitHub CI

GitHub CI is optional and should be offered during `/pm0 init`.

V1 CI should be warning-oriented.

- infer likely surfaces from changed paths, file names, PR text, `.pm0/surfaces/`, proposals, and PRDs
- product-changing PRs mention a PM0 proposal or PRD
- linked proposal/PRD file exists
- linked surface file exists
- PR explains product intent or links the relevant PM0 artifact

If the GitHub environment has an LLM review mechanism available, the CI can use it to infer affected surfaces and product-memory contradictions. Otherwise, it should fall back to static artifact checks.

Do not expose CI as `/pm0 review` in v1.

## README Expectations

The public repository should include a README inspired by Superpowers and Impeccable packaging:

- what PM0 is
- why repo-native product memory matters
- install instructions for supported harnesses
- basic command examples
- `.pm0` folder contract
- founder workflow examples
- note that PM0's implementation follows the local Superpowers and Impeccable example patterns
- optional GitHub CI setup

## Non-Goals

PM0 v1 should not include:

- source ingestion pipelines
- a SaaS dashboard
- a canvas UI
- a Linear/Jira replacement
- a Notion replacement
- a generic PRD generator
- separate decision logs

## Implementation Decisions

1. Target all practical harnesses represented by the Impeccable example pattern. Keep one canonical PM0 skill source, then produce harness-specific copies for directories like `.agents/`, `.claude/`, `.cursor/`, `.gemini/`, `.github/`, `.kiro/`, `.opencode/`, `.pi/`, `.qoder/`, `.rovodev/`, `.trae-cn/`, and `.trae/`, plus plugin manifests where the harness supports them.
2. Put deterministic file creation in `scaffold-pm0.mjs`: create `.pm0/project.md`, `.pm0/contexts/`, `.pm0/surfaces/index.md`, `.pm0/proposals/`, and `.pm0/prds/`. Keep product judgment in command instructions: which contexts to create, what assumptions to mark, which surfaces matter, and what evidence to use.
